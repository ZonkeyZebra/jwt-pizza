# Incident: 2026-04-07 12-07-25

## Summary
Between approximately 12:04 UTC and 12:20 UTC on 04/07/2026, users encountered consistent failures when attempting to place pizza orders on the JWT Pizza application.

The incident was triggered by an intentional chaos event injected into the Pizza Factory vendor service, which caused all downstream order fulfillment requests to fail with the message “Failed to fulfill order at factory.”

This was a Severity 2 incident, as ordering functionality was unavailable for a portion of users during the event window. The impact lasted approximately 20 minutes, after which the system recovered automatically once the chaos event ended.

The issue was detected through application logs and monitoring systems, and the team investigated by analyzing error logs and the vendor chaos report.

## Detection
This incident was detected through elevated error logs on the `/api/order` endpoint, which showed repeated failures with the message indicating factory fulfillment issues.

While logs clearly showed the issue, alerting was insufficient, as no proactive alert was triggered based on error rate thresholds. The issue required manual observation of logs to identify.

Improvements will include:
- Adding alerts for high error rates on critical endpoints
- Improving alert sensitivity and response coverage

## Impact
For approximately 10 minutes between 12:04 UTC and 12:20 UTC on 04/07/2026, users experienced failures when attempting to place pizza orders.

This incident affected an estimated 100% of users attempting to place orders during the window, as all requests to the Pizza Factory service failed. Users experienced failed order submissions and an inability to complete purchases.

No support tickets or social media reports were recorded, but the issue would have significantly impacted user experience during the incident window.

## Response
After observing failures around 12:07 UTC, the on-call engineer began investigating logs via Grafana at approximately 12:09 UTC.

The engineer identified that all failures were related to the downstream Pizza Factory service. The chaos report link included in the logs was used to further diagnose the issue as a chaos-induced failure.

Because the failure originated from an external vendor system under chaos testing, no code changes were made. The team monitored the system until the chaos event ended and service was restored.

Escalation occurred internally when additional support was requested to confirm the root cause and validate that no application-level fixes were required.

## Root cause
The root cause of the incident was an intentional chaos injection into the Pizza Factory vendor service, which caused downstream order fulfillment requests to fail. This was not caused by a bug in the application code, but rather by a simulated failure scenario designed to test system resilience and observability. All order requests failed because the application depends on the external Pizza Factory service, which was temporarily unavailable or degraded during the chaos event.

This chaos test did simulate what could occur if too many pizzas were being ordered in one go to demonstrate how that would affect our systems.

## Resolution
The incident was resolved automatically once the chaos testing period ended and the Pizza Factory service returned to normal operation. No direct mitigation actions were required to restore service, as the failure was external and temporary.

This test did reveal steps that could be taken to improve our response to errors and issues in the future. Such steps include:
- Faster identification of failures
- Clearer alerting
- Other automated fallback strategies

## Prevention
Similar incidents could occur if:
- External vendor services become unavailable
- Network issues prevent communication with dependencies
- Rate limiting or throttling is applied

To prevent delayed detection in the future:
- Improve monitoring of third-party dependencies
- Add alerts specifically for downstream service failures
- Implement dashboards that clearly separate internal vs external failures
